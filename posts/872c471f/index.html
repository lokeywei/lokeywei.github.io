谈到Spring，我不由得回忆刚毕业那会儿的一次面试经验，面试官让我简单的谈一下Spring。虽然当时我已经有了一些项目经验，但是这个问题还是让我有点摸不着头脑。

什么是Spring？为什么要用Spring？怎么去用Spring？

Spring的由来

Spring从何而来？

咱们为什么要讨论这个问题？特别在企业模块化开发向普遍化发展的今天，你可以不用了解数据库连接原理，线程池，资源池和网络通信知识，只要你会复制粘贴，会模仿demo写点代码就可以很快的投入到JAVA企业化开发当中。当然这种现象也是JAVA EE发展目标，毕竟JAVA平台推出伊始的目标就是为了方便程序员开发，也正是当时那句‘write once，run anywhere’吸引了无数开发者投身其中，也奠定了JAVA今时今日在企业化开发的地位。但是当我们真正回顾那段历史，我们不禁会为前人的智慧感到惊叹，会被那些令人拍案叫绝的思想深深震撼。

JAVA EE

众所周知，JAVA不仅仅是一门编程语言，还是包括JVM的JAVA平台的总称。

JAVA包含三种体系

  • JAVA SE （JAVA平台标准版）
  • JAVA EE (JAVA平台企业版)
  • JAVA ME（JAVA平台微型版）

咱们这里主要讨论一下JAVA平台企业版，也就是我们常说的JAVA EE。

JAVA EE与其说它是企业版平台，不如说它是一组规范化企业开发服务，它是命名和目录服务JNDI、RMI 、标准事务管理接口JTS和JTA、JCA 、资源池、线程管理的集合体。

提到JAVA EE就不可避免的提到EJB，现在打开百度搜索JAVA EE还是可以看到很多网站都会说JAVA EE的核心就是EJB，当然这些文章的日期都停留在2010年以前。很遗憾由于年龄的原因我没有经历过在EJB出生发展到日渐衰落的年代，这里推荐这本《Expert one-on-one J2EE development without EJB》，感谢这本书让我了解到那段我没接触到的历史。

Why without EJB？

这里借用一下这本书的标题，和JAVA平台一样，脱胎于JAVA EE的EJB框架在诞生之时也描述出了一幅令人向往的愿景。但是从今日的结果上来看，它并没有实现当初的承诺，尽管我没有使用过EJB进行开发，但是在我的印象当中这是一个重量级的框架，这个重量级是相对我们如今使用的轻量级框架来言的。

书中的观点我这里也不再赘述，有兴趣的同学可以去翻翻这本书。接下来我们进入正题，什么是Spring？

AOP与IOC

或许可以这么说，Spring就是在传统JAVA EE规范的基础上，通过IOC与AOP简化传统EJB框架的开发流程，以一种更友好的方式让程序开发者只关心业务的逻辑而不用考虑底层是如何运行的。

IOC容器

Spring用于管理Bean对象和对象之间关系的容器，被称为IOC容器。

BeanFactory

BeanFactory是一个接口，主要功能是实现依赖注入。

什么叫依赖注入？

在一个类中创建另外一个类，这种硬编码同通常是不被推荐的。

把类的创建交给BeanFactory，在需要使用的时候直接通过注解或者其他形式调用，这就是依赖注入。

典型的使用就是@Service，Spring项目启动的时候会把有带有@Service的类包装成Bean对象，这样不仅省去了本来需要构建的单例模式，而且在我们调用的时候只要通过@Autowired进行依赖注入就可以直接调用Service定义好的功能。

ApplicationContext

ApplicationContext是BeanFactory的子类，在BeanFactory的基础上增加了一些功能，通常在轻量级的应用中会选择使用BeanFactory。

ApplicationContext和BeanFactory一样在应用的最多的地方主要是对XML文件的解析，区别就在于XML
文件来源不同

  • FileSystemXmlApplicationContext 完整文件的路径
  • ClassPathXmlApplicationContext CLASSPATH目录下
  • WebXmlApplicationContext Web应用程序内
Bean的作用域
  • singleton 单例模式，只存在一个Bean
  • prototype 每次从容器中调用Bean时都会创建一个新的Bean
  • request 每次请求都会创建一个新的Bean
  • session 每次会话都会创建一个新的Bean

另外，在Bean的创建和销毁时，可以通过配置内部init和destory方法，来进行一些初始化和销毁操作。

AOP

AOP减少了侵入式编程，至少现在的业务层只用实现自己的业务而不用关心是否要校验当前用户权限。

AOP把我们从对象的关系中解放出来，从一个更高的层面去分析整个项目结构。

我们可以自定义Pointcut，可以是文件结构，同样也可以是一个注解，这就意味着我们可以把它放到任何一个我们想放到的地方。

我们可以在切入点的任意位置执行想要的advice，无论是之前还是之后，甚至可以在之前之后都执行。
